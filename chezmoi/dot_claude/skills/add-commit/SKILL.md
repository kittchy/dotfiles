# Gitで変更をステージング（add）してコミット

## ワークフロー

### 1. 変更状況の確認

`git status` と `git diff` で現在の変更内容を把握する。

### 2. ステージング（git add）

- 変更内容を確認し、コミット対象のファイルを選択してステージングする
- `git add <ファイルパス>` で個別にファイルを追加する（`git add .` や `git add -A` は使わない）
- 関連する変更をまとめて1つのコミットにする。無関係な変更が混在する場合は、別々のコミットに分ける
- `.env`、credentials、秘密鍵などの機密ファイルはステージングしない。ユーザーが明示的に指定した場合は警告する

### 3. コミットメッセージの作成

変更内容に応じたprefixを付けてコミットメッセージを作成する。

**Prefix一覧:**

| Prefix | 用途 |
|--------|------|
| `[FIX]` | バグや不具合の修正 |
| `[FEAT]` | 新機能の追加 |
| `[DOC]` | ドキュメントの更新や改善 |
| `[STYLE]` | コードフォーマットの修正、UIのみの変更 |
| `[REFACT]` | パフォーマンスの改善なしのコードの改善 |
| `[TEST]` | テストの追加や改善 |
| `[PREF]` | パフォーマンスの改善 |
| `[CHORE]` | ビルドプロセスの変更や改善 |
| `[WIP]` | 作業中 |
| `[DEL]` | コードの削除 |

**コミットメッセージの形式:**

```bash
git commit -m "[PREFIX] 変更内容の簡潔な説明"
```

**Example 1:**
Input: ログイン時のバリデーションエラーを修正
Output: `git commit -m "[FIX] ログイン時のバリデーションエラーを修正"`

**Example 2:**
Input: ユーザープロフィール画面を追加
Output: `git commit -m "[FEAT] ユーザープロフィール画面を追加"`

**Example 3:**
Input: 不要になったレガシーAPI関連コードを削除
Output: `git commit -m "[DEL] レガシーAPI関連コードを削除"`

### 4. コミット後の確認

`git status` でコミットが正常に完了したことを確認する。
