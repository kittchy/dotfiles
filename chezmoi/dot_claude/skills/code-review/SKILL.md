# コードレビュー（Context7 + Codex統合）

Context7のドキュメント検索とcodex（利用可能な場合）を活用して、プロジェクトのコードを包括的にレビューするスキル。

## レビュー実行フロー

1. **ツール確認**: `command -v codex` でcodexの利用可否を判定
2. **プロジェクト分析**: 依存関係・ライブラリを識別
3. **コード分析**: codex（利用可能時）または標準ツールでコードパターンを分析
4. **ドキュメント照合**: Context7で各ライブラリのベストプラクティスを取得
5. **レビュー生成**: 改善提案を優先度付きで出力

詳細な実行手順は `examples/review-process.md` を参照。

## ツールガイド

- **Context7の使い方**: `references/context7-guide.md` を参照
- **Codexの使い方**: `references/codex-guide.md` を参照（フォールバック方法も記載）

## レビュー観点

### 1. ベストプラクティス準拠
- Context7で取得した公式ドキュメントのベストプラクティスとの照合
- 推奨パターンと実装の比較
- 非推奨APIや古い手法の検出

### 2. コード品質
- 可読性: 命名規則、コメント、構造
- 保守性: モジュール化、重複コード、複雑度
- 一貫性: プロジェクト内のコーディングスタイル

### 3. セキュリティ
- 一般的な脆弱性パターン（OWASP Top 10）
- 入力検証とサニタイゼーション
- 認証・認可の実装

### 4. パフォーマンス
- 非効率なアルゴリズムやクエリ
- メモリリークの可能性
- 不要な再レンダリングや再計算

### 5. エラーハンドリング
- 適切な例外処理
- エラーメッセージの品質
- リソースのクリーンアップ

## 注意事項

- **codex**: レビュー開始時に必ず存在確認する。利用不可の場合はgrep/find/標準ツールにフォールバック
- **Context7**: 同じライブラリのドキュメント検索は最大3回まで。ライブラリID解決はライブラリごとに一度のみ
- **機密情報**: Context7のクエリにAPIキー・パスワード・個人情報を含めない
- **フィードバック**: 問題点だけでなく良い点も評価し、具体的な改善案と優先度を提示
- **既存パターン尊重**: 過度なリファクタリングを提案しない。実際の影響を評価してから提案

## 出力形式

- 日本語で記述
- コードスニペットと改善例を含める
- ファイル名と行番号を明記（例: `src/app.ts:42`）
- 使用した分析ツール（codex / 標準ツール）をレビュー概要に記載
- Context7で取得した情報の出典を示す
- 出力テンプレートは `examples/review-process.md` のステップ4を参照
